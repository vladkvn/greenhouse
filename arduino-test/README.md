# Arduino test: датчик + LCD + ESP8266 (Wi‑Fi)

Два скетча: **Arduino Uno** снимает показание с **A0**, выводит на **LCD I²C (0x27)** и обменивается строками по UART с **модулем ESP8266** (Petoi). ESP подключается к домашней Wi‑Fi и даёт **HTTP API**: телеметрия и команды.

## Схема подключения

| ESP8266 | Arduino Uno |
|--------|-------------|
| **TX** | **D2** (приём `SoftwareSerial`, RX) |
| **RX** | **D3** (передача с Uno; **обязателен делитель 5 В → 3,3 В** на эту линию) |
| **GND** | **GND** |

Питание ESP — по документации модуля (3,3 В, достаточный ток). Общая земля с Uno обязательна.

## Протокол по UART (**9600** бод, строки с `\n`)

| Направление | Формат | Назначение |
|-------------|--------|------------|
| Uno → ESP | `T,<число>` | Телеметрия (число — показание АЦП, обычно 0…1023) |
| ESP → Uno | `I,<IPv4>` | После Wi‑Fi — повтор каждые **~4 с** в течение **~2 мин**, чтобы Uno не пропустил строку; LCD строка 2 до первой команды |
| ESP → Uno | `C,<ИМЯ>` | Команда из whitelist (см. ниже) |

Пустые и нераспознанные строки игнорируются.

### Разрешённые команды (одинаково в обоих скетчах)

- `PING` — отладка (в Serial Uno печатается `cmd:PING`).
- `RELAY_ON` / `RELAY_OFF` — виртуальное «реле» (флаг на Uno, на LCD строка 2: `R:ON` / `R:OFF`). Подключите реальное реле к цифровому пину и `digitalWrite` в обработчике команды при необходимости.

## Настройка Wi‑Fi на ESP8266

В `esp8266-web-bridge/esp8266-web-bridge.ino` задайте:

- `STASSID` — имя сети (SSID).
- `STAPSK` — пароль.

Либо при сборке передайте флаги компилятора `-DSTASSID=...` и `-DSTAPSK=...`.

**Не коммитьте** реальные пароли в открытый репозиторий.

В прошивке ESP: встроенный светодиод (обычно GPIO2) — **горит** при подключении к Wi‑Fi, **мигает**, пока сеть не найдена. Отдельный индикатор питания на плате может вести себя иначе.

Отладка Wi‑Fi: USB к **ESP**, Serial Monitor **9600** — строки с префиксом **`[wifi]`** (статус, SSID, DHCP, RSSI, обрывы). Пароль сети в лог не выводится.

### Опционально: токен для `/command`

Если задать непустой `COMMAND_TOKEN`, каждый запрос к `/command` должен содержать параметр `token` с этим значением (иначе `401`).

## Прошивка

1. **ESP8266** — в Arduino IDE: менеджер плат ESP8266, плата **Generic ESP8266 Module** (как в [документации Petoi](https://docs.petoi.com/communication-modules/wifi-esp8266)). Откройте каталог `esp8266-web-bridge/`, залейте `esp8266-web-bridge.ino` через USB‑адаптер Petoi.
2. **Uno** — откройте `arduino-test.ino`, выберите плату Arduino Uno, загрузите.

### Проверка только Wi‑Fi (Uno + ESP)

Скетч **`arduino-wifi-test/arduino-wifi-test.ino`** (на Uno, ESP с `esp8266-web-bridge`): ждёт строку `I,<IP>` по UART и пишет в Serial `[OK]` или таймаут. Без LCD и датчика.

После подключения к Wi‑Fi ESP шлёт на Uno строки `I,<IP>` (см. таблицу выше). Адрес можно увидеть на **второй строке LCD** (см. ниже). На Uno в USB Serial при успешном разборе IP печатается строка `ip:<адрес>` (**9600** бод).

### «Квадратики» в Serial Monitor и разные скорости

USB‑Serial и линия Uno↔ESP — **разные вещи**:

| Куда подключён USB | Скорость монитора | Что должно быть читаемо |
|--------------------|-------------------|-------------------------|
| **Arduino Uno** | **9600** | Строки `T,...`, при приходе IP с ESP — `ip:192.168.x.x` |
| **ESP8266** (адаптер Petoi) | **9600** | Строки `I,192.168.x.x` (и то же на линии к Uno) |

Если выставить **57600** или **115200**, глядя на **Uno**, вы увидите «квадратики» — это **не** ошибка прошивки, а **несовпадение baud**. Первые доли секунды после сброса ESP загрузчик может слать на другой скорости — мусор в начале допустим.

### Нет IP на дисплее

- Проверьте **TX ESP → D2**, **D3 → RX ESP**, общий **GND**, **делитель 5 В→3,3 В** на линии с D3.
- Перешейте **оба** скетча после смены скорости на **9600**.
- Дайте **до ~2 минут** после появления Wi‑Fi у ESP — строка `I,...` повторяется.

## HTTP API (в одной LAN с ПК)

Подставьте IP ESP8266 (из Serial или админки роутера).

### Телеметрия

```http
GET http://<IP>/sensor
```

Ответ `200`:

```json
{"value":512,"ageMs":120}
```

`ageMs` — сколько миллисекунд прошло с последней принятой по UART строки `T,...`. Если Uno ещё не прислал данные — `503` и `{"error":"no_data_yet"}`.

Корень `GET /` перенаправляет на `/sensor`.

### Команда

```http
GET http://<IP>/command?cmd=RELAY_ON
```

или то же с `RELAY_OFF`, `PING`.

Если включён `COMMAND_TOKEN`:

```http
GET http://<IP>/command?cmd=PING&token=ВАШ_СЕКРЕТ
```

Ответ `200`:

```json
{"ok":true,"cmd":"RELAY_ON"}
```

Неверное или пустое имя команды — `400`. Без токена при включённом токене — `401`.

### Примеры с терминала (curl)

```bash
curl -s "http://192.168.1.50/sensor"
curl -s "http://192.168.1.50/command?cmd=RELAY_ON"
curl -s "http://192.168.1.50/command?cmd=RELAY_OFF"
curl -s -X POST -d "cmd=PING" "http://192.168.1.50/command"
```

Замените `192.168.1.50` на IP вашего ESP8266.

## Сервер GreenHouse (Nest API, `apps/api`)

ESP может **сам** регистрироваться в бэкенде и слать показания раз в минуту (HTTP **клиент** к вашему ПК/серверу, где запущен Nest + PostgreSQL).

В `esp8266-web-bridge/esp8266-web-bridge.ino` задайте:

| Макрос | Назначение |
|--------|------------|
| `GH_API_HOST` | IP или hostname машины с API (например `192.168.1.10`). Пустая строка — **отключить** исходящие запросы к Nest. |
| `GH_API_PORT` | Порт (по умолчанию `3000`). |
| `GH_API_KEY` | Тот же секрет, что **`API_KEY`** в `apps/api/.env`. |
| `GH_DEVICE_ID` | Совпадает с `deviceId` в API (по умолчанию `gh-node-1`). |
| `GH_READINGS_INTERVAL_MS` | Интервал отправки `POST /readings` (по умолчанию `60000`). |

После прошивки в Serial ESP (9600) будут строки **`[api]`**: `register http=...`, затем периодически `readings http=...`. Код ответа `2xx` — успех.

Телеметрия в теле запроса: `{"moisture":<последнее T,... от Uno>}` — поле согласовано с `POST /readings` в `apps/api`.

**Не коммитьте** реальный `GH_API_KEY` в открытый репозиторий; удобнее задавать макросы через флаги сборки (`-DGH_API_HOST=...` и т.д.).

## Поведение на LCD (Uno)

- Строка 1: `S:<значение>` — показание с A0.
- Строка 2 **до первой команды** с ESP: сначала **`WiFi...`**, после прихода строки `I,<IP>` с модуля — **адрес IPv4** (чтобы открыть `http://<IP>/sensor` с ПК в той же сети).
- После **первой** принятой команды (`C,...`) вторая строка переключается на обычный режим: `WET` / `DRY` и `R:ON` / `R:OFF`.
- После любой команды на **~2,5 с** вторая строка показывает **`Cmd <имя>`**, затем снова `WET`/`DRY` и реле.

## Расширение

- Добавьте новые имена в **whitelist** в обоих файлах и реализацию на Uno (пины, ШИМ и т.д.).
- Для надёжности в продакшене ограничьте доступ к HTTP (отдельная VLAN, firewall, сильный `COMMAND_TOKEN`).
